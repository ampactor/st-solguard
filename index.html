Hook failure recovery can bypass payment verification
Disputed Low

The FacilitatorHooks system allows on_verify_failure and on_settle_failure hooks to return FailureRecovery::Recovered(response), completely replacing a verification error with a fabricated success. A poorly written or malicious hook can intercept any verification failure and return VerifyResponse::valid(fake_payer), bypassing all security checks. Since hooks are composed at the application layer via with_hook()/add_hook(), SDK users integrating third-party hooks could unknowingly compromise the verification pipeline.

Fix: Separate recovery hooks from observation hooks. Add explicit authorization flag for recovery capability. Log warnings whenever FailureRecovery::Recovered is returned.

Source: x402 Payment Protocol: AI-Agent Micropayment Rails on Solana (75%) → target selection → deep agent review → Disputed
Validation reasoning

The code at hooks.rs:246-254 confirms that on_verify_failure hooks can return FailureRecovery::Recovered(response) which replaces an error with a success. However, the severity is overstated: (1) This is an intentional SDK design pattern mirroring the official x402 Go SDK (hooks.rs:17, 94-99, 168). The hooks system is explicitly designed for extensibility, and recovery is a documented capability. (2) Hooks are registered at compile-time via with_hook() or add_hook() — they are not runtime-configurable or user-supplied. The SDK integrator writes and registers hooks in their own code. (3) The 'third-party hooks' risk scenario is speculative — there is no plugin ecosystem or dynamic hook loading. An integrator importing a malicious crate has far worse problems than hook bypass. (4) The FacilitatorHooks trait defaults to FailureRecovery::Propagate (hooks.rs:129), so recovery requires explicit implementation. This is a design decision, not a vulnerability. Any Facilitator impl could be malicious — the hooks system doesn't introduce new attack surface beyond what already exists by implementing the trait. This should be Low/Informational at best — it's documenting expected SDK behavior, not a security flaw.

Confirmed High

The QuoteVerifier::verify() function reads the queue account to extract authorized oracle signing keys, but performs no owner check (account.owner == SWITCHBOARD_PID) and no discriminator check on the queue account. Oracle Index Silent Modulo Wrapping Allows Index Aliasing
Confirmed Medium

In verify(), oracle indices from the ed25519 instruction data are reduced modulo 30 to bound array access: (oracle_idxs[i] as usize) % 30. This is labeled 'branchless bounds check' but it silently maps any oracle index to a valid array position. Index 5 and index 35 both resolve to position 5. This means the oracle_idx value in the signed message does not need to be the canonical index — multiple values map to the same oracle key. While the ed25519 signature itself is verified against the precompile, the oracle index is part of the unsigned suffix data (extracted at lines 180-188 from the end of instruction data, outside the signed message). An attacker who can modify the oracle_idxs suffix without invalidating the ed25519 signature could alias oracle indices.

Unsound SlotHash Slice Transmute — Metadata Reinterpretation
Disputed Low

In find_slothash_in_sysvar(), a &[u8] slice (slothash_data[8..]) is transmuted to &[SlotHash] using std::mem::transmute. This is undefined behavior: transmute on a fat pointer (&[u8] is (ptr, byte_len)) reinterprets the byte length as the element count of &[SlotHash]. A 20480-byte slice becomes a &[SlotHash] with len=20480 instead of len=512 (20480/40). Any indexing above 512 would read out of bounds. The estimated_idx calculation uses modulo SYSVAR_SLOT_LEN (512), which bounds it correctly in practice, but the slice metadata itself is wrong — slot_data.len() returns 20480 not 512, and any code that trusts the slice length would read beyond the buffer.
